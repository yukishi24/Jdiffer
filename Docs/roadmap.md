# Product Roadmap

## 1. このロードマップの前提

- 詳細な日付は書かない
- 内容は変わってよい
- Sprint Planning より優先度は低い
- 判断に迷ったときの「軸」として使う

---

## 2. フェーズ定義

ロードマップはフェーズ単位で管理する。

例：

- Phase 0：準備
- Phase 1：MVP
- Phase 2：改善
- Phase 3：拡張

※ フェーズ数は増やしすぎない（最大4）

---

## 3. 全体ロードマップ

### Phase 0：準備（必要な場合）

※ 技術・構成に迷いがある場合のみ実施  
※ 長くやらない（1 Sprint 以内）

**目的**

- 開発を迷わず進められる状態にする

**やること**

- プロダクトビジョンの整理
- 技術スタック決定
- リポジトリ構成の確定
- 最初の User Story 作成
- 技術的な不確実性の調査（Spike）

**やらないこと**

- 本格的な機能実装
- UIの作り込み
- 最適化・設計の作り込み

**完了条件**

- 「これで作り始める」と判断できる状態

---

## 4. 優先順位の考え方

判断に迷ったら以下の順で考える。

1. プロダクトビジョンに合っているか
2. 今のフェーズに合っているか
3. User Story として説明できるか
4. 今やらない理由はあるか

---

## 5. Sprint との関係

- Roadmap → 直接 Issue は切らない
- Roadmap → User Story の種を生む
- Sprint Backlog は **Roadmapより常に優先**

---

## 6. 更新ルール

- Sprint 中は更新しない
- Sprint 境界でのみ更新する
- 変更したら理由を残す
