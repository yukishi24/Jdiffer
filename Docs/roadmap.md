# Product Roadmap

このロードマップは  
**中長期の方向性を共有・確認するためのもの**。

Sprint の計画や Issue の順序を直接縛るものではない。

---

## 1. このロードマップの前提

- 詳細な日付は書かない
- 内容は変わってよい
- Sprint Planning より優先度は低い
- 判断に迷ったときの「軸」として使う

---

## 2. フェーズ定義

ロードマップはフェーズ単位で管理する。

例：

- Phase 0：準備
- Phase 1：MVP
- Phase 2：改善
- Phase 3：拡張

※ フェーズ数は増やしすぎない（最大4）

---

## 3. 全体ロードマップ

- 開発・検証を始められる状態にする

**やること**

- プロダクトビジョン策定
- 技術スタック決定
- 最初の User Story 作成

**やらないこと**

- 機能実装
- UI の作り込み

## 4. 優先順位の考え方

判断に迷ったら以下の順で考える。

1. プロダクトビジョンに合っているか
2. 今のフェーズに合っているか
3. User Story として説明できるか
4. 今やらない理由はあるか

---

## 5. Sprint との関係

- Roadmap → 直接 Issue は切らない
- Roadmap → User Story の種を生む
- Sprint Backlog は **Roadmapより常に優先**

---

## 6. 更新ルール

- Sprint 中は更新しない
- Sprint 境界でのみ更新する
- 変更したら理由を残す
